## 目标

依次执行 **收集 → 分析 → 生成 → 演进** 四阶段，形成完整的知识演进闭环，持续优化 Skill 设计能力。

---

## 输入

### 必需输入
- 无特定前置条件，可从任意阶段启动

### 可选输入
- `--round=N`：指定演进轮次编号（默认自动递增）
- `--stage=<阶段名>`：从指定阶段开始（collect/analyze/generate/evolve）
- `--dry-run`：仅模拟执行，不写入文件

---

## 四阶段流程

```

│   收集阶段   │───>│   分析阶段   │───>│   生成阶段   │───>│   演进阶段   │

      │                  │                  │                  │
      v                  v                  v                  v
 data/collected/    data/analyzed/    data/generated/    memory/ + docs/
```

### 依赖关系

| 阶段 | 前置依赖 | 核心输出 | 确认机制 |
|------|----------|----------|----------|
| 收集 | 无 | `data/collected/*.md` | 检查文件数量 ≥ 1 |
| 分析 | 收集阶段完成 | `data/analyzed/pattern-summary.md` | 检查范式数量 ≥ 1 |

## 执行步骤

### 阶段 1：收集（Collect）

**触发方式**：调用 `src/collect-skills.md`

**执行命令**：
```
/Skill skill=collect-skills
```

**执行流程**：
1. 检查 `data/collected/` 目录状态
2. 执行收集 Skill，从 skillhub.cn 获取热门 Skill
3. 等待收集完成

**阶段确认**：
```bash
# 检查是否有新收集的 Skill
ls -la data/collected/*.md | wc -l
```

**确认标准**：
- [ ] 至少收集 1 个 Skill 文件
- [ ] `data/collected/index.md` 已更新
- [ ] 每个 Skill 文件包含完整 frontmatter

**失败处理**：
- 如确认失败，记录错误并终止流程
- 用户可手动补充后重试：`/self-evolve --stage=analyze`

---

### 阶段 2：分析（Analyze）

**触发方式**：调用 `src/analyze-patterns.md`

**执行命令**：
```
/Skill skill=analyze-patterns
```

**执行流程**：
1. 读取 `data/collected/` 下的 Skill 文件
2. 执行五维度分析（元数据、结构、内容、触发、质量）
3. 识别设计范式并归类
4. 生成分析报告

**阶段确认**：
```bash
# 检查分析报告是否存在
test -f data/analyzed/pattern-summary.md && echo "PASS" || echo "FAIL"
```

**确认标准**：
- [ ] `data/analyzed/pattern-summary.md` 已生成
- [ ] 至少识别 1 类结构范式
- [ ] 至少识别 1 类内容范式
- [ ] 质量评估有明确评分依据

**失败处理**：
- 如确认失败，回退到收集阶段重新执行
- 记录分析过程中的异常到 `logs/analyze-errors.log`

---

### 阶段 3：生成（Generate）

**触发方式**：调用 `src/generate-meta.md`

**执行命令**：
```
/Skill skill=generate-meta
```

**执行流程**：
1. 读取 `data/analyzed/pattern-summary.md`
2. 提取关键范式并归类
3. 生成 Skill 模板（简单/工作流/复杂）
4. 创建质量检查清单
5. 生成元技能 Skill 文件

**阶段确认**：
```bash
# 检查元技能模板是否生成
test -f data/generated/skive-template.md && echo "PASS" || echo "FAIL"

# 检查模板目录是否完整
ls data/generated/skill-templates/ | wc -l
```

**确认标准**：
- [ ] `skive-template.md` 已生成
- [ ] 至少 3 种模板类型已创建
- [ ] `quality-checklist.md` 已生成
- [ ] 模板内容符合范式规范

**验证测试**：
- 使用生成的模板创建测试 Skill
- 运行质量检查清单验证
- 确认测试 Skill 可正常触发执行

**失败处理**：
- 如模板生成失败，检查分析报告完整性
- 如质量检查失败，记录不通过项并人工复核

---

### 阶段 4：演进（Evolve）

**触发方式**：本阶段直接执行，不调用外部 Skill

**执行流程**：

#### 4.1 更新设计原则文档

**目标文件**：`docs/design-principles.md`

**更新规则**：
1. 如发现新的重要设计原则，追加到文档
2. 保持原则编号连续
3. 标注发现来源和日期

**更新格式**：
```markdown
## N. {原则名称}

**发现来源**：{演进轮次} - {Skill 名称} 分析

**发现日期**：{YYYY-MM-DD}

**原则内容**：{原则描述}

**应用场景**：{何时应用}

**示例**：
- 正例：{做法}
- 反例：{做法}
```

#### 4.2 更新趋势分析文档

**目标文件**：`docs/trend-analysis.md`

**更新条件**：发现新的范式趋势或设计模式变化

**更新内容**：
- 新增范式类型统计
- 使用频率变化趋势
- 设计偏好演变

#### 4.3 记录演进记忆

**目标目录**：`memory/`

**存储内容**：本轮演进的关键发现摘要

**记忆格式**：

## 触发机制

### 手动触发

**命令**：
```
/self-evolve
```

**可选参数**：
- `--round=N`：指定轮次
- `--stage=<阶段>`：从指定阶段开始
- `--dry-run`：模拟执行

**适用场景**：
- 初次执行自演进
- 需要立即更新知识资产
- 手动修复失败的演进轮次
- 测试新配置

### 定期触发

**配置方式**：通过 `/schedule` 命令配置定时任务

**推荐配置**：
```
# 每周一凌晨执行自演进
/schedule create --cron="0 2 * * 1" --prompt="/self-evolve"
```

**定时触发参数**：
| 触发频率 | Cron 表达式 | 适用场景 |
|----------|-------------|----------|
| 每日 | `0 2 * * *` | 高频迭代项目 |
| 每周 | `0 2 * * 1` | 标准演进节奏 |
| 每两周 | `0 2 1,15 * *` | 稳定维护阶段 |
| 每月 | `0 2 1 * *` | 低频更新需求 |

**定时执行限制**：
- 检查上次演进时间，避免重复执行
- 如距离上次演进不足 24 小时，跳过执行
- 记录跳过原因到 `logs/evolution/skipped.log`

## 异常处理

### 阶段执行失败

**处理策略**：
1. 记录失败阶段和错误信息
2. 终止后续阶段执行
3. 输出失败报告
4. 用户可指定 `--stage` 从失败阶段重试

### 数据不一致

**检查项**：
- 收集的 Skill 数量与索引记录是否匹配
- 分析报告引用的 Skill 是否存在
- 生成的模板是否符合范式规范

**修复方式**：
- 重新执行不一致的阶段
- 手动修正索引文件

### 网络访问失败

**影响阶段**：收集阶段

**处理方式**：
1. 重试最多 3 次
2. 使用缓存的历史数据（如有）
3. 跳过收集，使用已有数据执行后续阶段

## 注意事项

### 执行原则

1. **阶段完整性**：每个阶段必须确认完成后再进入下一阶段
2. **数据可追溯**：所有输出文件标注来源和时间
3. **失败可恢复**：支持从任意阶段重新开始
4. **日志详尽**：记录每个关键操作的时间戳和结果

### 质量保证

1. 演进后人工复核关键发现
2. 定期清理过期或低质量的收集数据
3. 保持设计原则文档的精简和实用性
4. 避免重复收录相同 Skill

### 持续优化

1. 定期评估演进效率
2. 优化阶段间的数据传递
3. 更新分析维度以适应新范式
4. 调整定时触发频率以匹配项目需求

## 后续步骤

演进完成后：
1. 查看演进日志了解关键发现
2. 复核设计原则文档的更新内容
3. 使用生成的模板创建新 Skill
4. 根据待改进事项规划下一步工作
5. 可继续执行下一轮演进形成持续循环
